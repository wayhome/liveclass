Reverse Engineering Bambu Connect

作者: pabs3 | 评分: 278 | 评论数: 18 | 发布时间: 2025-01-20 03:08
文章摘要

本文介绍了如何逆向工程Bambu Connect应用程序以提取私钥和证书。Bambu Connect是一个基于Electron的应用程序，采用隐蔽式安全机制，安全性较低。通过分析其main.js文件，可以找到存储在应用程序中的私钥。具体步骤包括：从MacOS的.dmg文件中提取应用文件，使用Ghidra工具找到GetKey函数以获取加密密钥，然后使用asarfix工具修复加密的app.asar文件，最终提取并解密main.js中的私钥和证书。文章还提供了用于解密过程的Python代码示例，帮助用户重新实现提取过程。

UK's hardware talent is being wasted

作者: sebg | 评分: 251 | 评论数: 43 | 发布时间: 2025-01-19 23:52
文章摘要

文章指出，英国顶尖大学如帝国理工、牛津和剑桥培养了世界一流的硬件工程师，但他们的职业发展面临严峻的经济现实。硬件工程师的起薪远低于软件工程师或金融业岗位，导致大量人才流失至咨询、金融或海外市场。这种人才浪费不仅体现在工资差距，还影响了国家创新能力和经济效益。原因包括地理限制、风投资本不足和传统工程公司缺乏创新。文章呼吁抓住当前机会，通过有雄心的硬件创业项目吸引和留住本地顶尖工程人才，避免进一步的“人才流失”。现在行动，英国有机会在全球硬件领域占据领先地位。

评论摘要
主要讨论点：英国职业选择、收入差距、行业发展与经济前景 不同观点： • **[职业选择与收入差距]** [lordnacho]、[refrigerator]、[tidenly] 等人指出，英国工程类工作的薪资较低，特别是在与金融、咨询和美国科技公司对比时。许多人因此转向了金融、咨询或软件行业，尤其是为了更高的收入和更好的职业发展。相比之下，传统工程行业如航空航天和材料工程的薪资难以与金融或科技行业相比。[refrigerator] 提到其伴侣在博物馆领域拥有丰富经验，但在英国的薪资远低于在美国获得的薪资，显示出非金融领域在英国的普遍低收入问题。 • **[英国创业与风险投资的困境]** [r_thambapillai]、[thom]、[alephnerd] 等人讨论了英国创业环境和风险投资的不足。他们指出英国缺乏像美国那样的高风险高回报投资文化，政府的治理不善和经济政策的失误导致了风险资本的匮乏。英国在吸引和留住高潜力科技企业（如DeepMind被谷歌收购）方面表现不佳，且缺乏像美国那样的创业生态系统。 • **[工程与制造的分离]** [Animats] 强调了工程与制造不能分离的观点，认为如果工程设计与实际制造过程脱节，会导致成本上升和设计不切实际。这一观点强调了实际制造经验在工程职业中的重要性，并指出英国和其他发达国家工程职位外包至低成本国家的趋势。 • **[全球普遍现象还是英国特有问题？]** [awanderingmind]、[pjmlp] 等人认为，英国的职业收入和选择问题并非独有，而是全球普遍现象，尤其是在硬件工程和软件工程的对比中。他们指出，即使在南非等国家，迁往英国从事金融行业仍被视为职业上的成功。此外，[pjmlp] 提到硅谷等地的科技行业也存在供需不匹配的问题，许多科技从业者从事着与他们技能不匹配的工作。 补充讨论： • **[地域与行业集中问题]** [tikkabhuna] 提出，英国缺乏像伦敦金融区那样的行业集中 hub，导致许多人在转换工作时不得不搬迁，增加了职业选择的地域限制。政府需要在交通、住房、教育等方面提供更多支持，以促进区域性行业中心的形成。 • **[文化和风险偏好差异]** [r_thambapillai]、[thom] 等人提到英国文化中缺乏像美国那样的高风险高回报的创业精神，更多人追求“稳定”的高薪工作，而不是冒险创业。英国的税收政策和房地产市场也被认为是影响风险资本和创业活力的因素。 争议焦点： • **[硬件与软件行业的对比]** [remus] 和 [lordnacho] 的讨论中，关于硬件行业是否比软件行业更具风险和挑战性存在争议。一方认为硬件涉及制造、物流等多重风险，而另一方则认为只要具备正确的商业策略和风险管理，硬件行业同样可以成功扩展。 • **[英国是否还有发展潜力]** [James_K] 对英国未来持悲观态度，认为国家正在走向衰落。而 [kleiba] 则认为，如果收入是主要目标，年轻人完全可以选择暂时移居国外以获取经验和更高收入，显示出对英国未来发展的不同看法。 总结：评论中主要围绕英国工程类职业的低薪与金融、咨询等行业高薪的对比，讨论了英国创业环境的不足、风险投资的缺乏以及地域和文化对职业选择的影响。争议点集中在硬件与软件行业的对比以及英国未来发展的潜力。

文章摘要

在1960年代至1980年代期间，阿尔巴尼亚在霍查（Enver Hoxha）领导下建造了超过75万个混凝土碉堡，平均每平方公里5.7个。这些碉堡遍布全国，从山间隘口到城市街道，旨在抵御潜在的外敌入侵，但从未实际用于该目的。碉堡的建设耗资巨大，导致国家资源紧张，无法解决住房短缺和道路问题。霍查去世后，阿尔巴尼亚于1987年才结束与希腊的战争状态，并在1992年共产主义政府解体后废弃了这些碉堡。部分碉堡在后来的1997年叛乱和1999年科索沃战争中被使用，如今多数被废弃，有些被改作住宅、咖啡馆、仓库或庇护所。阿尔巴尼亚的军事 doctrine 基于“人民战争”概念，强调全民防御，受二战期间阿尔巴尼亚抵抗运动影响。

It's time to make computing personal again

作者: mariuz | 评分: 481 | 评论数: 48 | 发布时间: 2025-01-19 23:11
文章摘要

I'll think twice before using GitHub Actions again

作者: nemwiz | 评分: 22 | 评论数: 7 | 发布时间: 2025-01-20 03:41
文章摘要

本文作者对GitHub Actions在处理大型单体仓库（monorepo）时的表现表达了不满，特别是在管理复杂的流水线和合并请求的检查时遇到了困难。以下是主要问题： 1. **合并请求和必要检查**：在单体仓库中，每个文件夹有独立的流水线，但GitHub要求为每个文件夹单独设置“必要检查”，导致如果某个PR没有触发的检查，就无法合并。 2. **可重用性和YAML复杂性**：随着流水线增长，管理变得复杂，需要大量条件语句（if）和重复代码。虽然可以拆分为多个工作流，但增加了维护负担。 3. **缺乏本地开发支持**：GitHub Actions无法在本地运行，现有工具（如act）体验不佳。 4. **GitHub的响应问题**：许多问题长期未解决，部分讨论线程被关闭，未见明显改进。 作者建议在CI/CD工具选择上考虑其他选项，如GitLab、Jenkins、TeamCity，或新兴工具如Dagger。

评论摘要
主要讨论点：GitHub Actions 的使用体验及其在不同场景下的适用性 不同观点： • [arghwhat] 认为不应在管道 DSL 中包含任何实际逻辑，只用于任务的编排。复杂的任务应通过仓库中的脚本来简化为一行命令。这种方法确保了任务可以在本地通过一行命令手动执行，增加了灵活性。 • [flohofwoe] 指出 GitHub Actions 的运行器速度很慢，使用体验不佳，感觉像是在非常慢的机器上运行构建。 • [androa] 表示 GitHub Actions 不适合用于 monorepos（单体仓库），并提到他们使用 `meta` 来模拟 monorepos，以减少缺点。 • [vrnvu] 反驳了一些对 GitHub 的批评，认为 GitHub 关心平台上的活跃用户，无论是管理 PR、代码审查还是查看失败动作的日志。 • [makingstuffs] 提到可以使用 `act` 工具在本地运行一些 GitHub Actions，质疑那些声称不能在本地运行 Actions 的观点。 • [tevon] 批评了在 GitHub Actions 中不断推送到分支以运行动作的模式，称之为“搜索并部署”，并希望至少有基本的健全性检查可以在本地进行。 • [bramblerose] 总结为这是在一个第三方平台上构建东西时遇到的经典问题，平台不会完全匹配个人用例。建议要么调整工作流以适应平台，要么选择降低到 IaaS 层，以获得更大的灵活性但失去一些特定功能。 补充讨论： • 争议的焦点在于 GitHub Actions 的速度、对 monorepos 的支持以及是否可以在本地运行 Actions。 • 一些用户建议通过脚本简化任务逻辑以增强灵活性，而另一些用户则对 GitHub Actions 的性能和适用性提出质疑。 • 讨论还涉及平台适应性问题，即在 PaaS 世界中，用户应调整工作流以适应平台的限制。
