gRPC vs. REST: Understand gRPC, OpenAPI and REST and When to Use in API Design

作者: hui-zheng | 评分: 128 | 评论数: 27 | 发布时间: 2025-01-23 00:47
文章摘要

本文讨论了API设计中gRPC、REST和OpenAPI的特点及适用场景。主要比较了两种API设计模型：RPC和REST。REST模型基于可寻址的资源，客户端无需理解URL格式；而RPC模型（如gRPC）基于可寻址的程序，数据隐藏在程序之后。 1. **REST**：较少使用，客户端直接使用服务器提供的URL，无需构造URL，简单直接，无需额外技术。 2. **gRPC**：使用HTTP/2作为传输协议，隐藏HTTP细节，客户端通过调用生成的存根（stub）来发起调用，不需了解底层HTTP映射。 3. **OpenAPI**：广泛用于设计RPC API，客户端需根据模板构造URL，并填入参数，与REST相反。 文中建议根据具体需求选择合适的API设计方式，REST适合简单场景，gRPC适合高性能、跨语言的场景，OpenAPI则适合需要明确URL构造的API设计。

评论摘要
主要讨论点：gRPC的优缺点及其适用场景 不同观点： • **反对gRPC**： - **jdwyah**：认为gRPC隐藏了内部复杂性，实际使用中会遇到许多问题，如调试困难、Maven插件问题、负载均衡不兼容HTTP2、文档差等，最终导致时间和精力的浪费。 - **bitzun**：指出gRPC在跨语言和跨团队使用时会遇到很多麻烦，通常需要一个中间层。尤其是在与非主流语言或旧版本语言对接时问题更多。 - **gghoop**：认为gRPC的性能并不突出，开源客户端质量差，尤其是在非核心语言（如Node.js）的实现上。对于数据中心内的使用不推荐gRPC。 - **rednafi**：认为gRPC的开发体验比REST差，需要额外的工具，生成的客户端代码质量低，且命令不如REST简单直接。 - **awinter-py**：警告不要陷入gRPC的生态系统，需要不断重新实现REST生态中已有的库，且调试和兼容性问题多，特别是对HTTP2支持不足。 - **est**：认为如果没有单代码库支持，维护.proto文件会非常困难，特别是在.proto文件复杂且开发者害怕更新时，不推荐使用gRPC。 • **支持gRPC**： - **jon_richards**：推荐使用connectrpc，解决了gRPC的许多问题，特别是自动生成SDK和更好的流支持。认为buf.build的依赖管理功能非常有用。 - **abalaji**：认为gRPC的生成消息定义和存根功能使得客户端请求更稳定，服务器响应更明确，从而使API集成更容易。 - **toprerules**：指出gRPC在FAANG公司内部服务路由中非常有用，特别是在需要快速和大规模服务间通信时。认为RPC协议在内部服务通信上比JSON/REST更有效。 • **中立/其他观点**： - **NAHWheatCracker**：个人经验中，gRPC因前端需求最终改用JSON API，但承认在工作之外使用gRPC时感觉不错，只是需要更多思考和工作量。 - **oppositelock**：认为OpenAPI和REST没有本质区别，OpenAPI只是描述HTTP API的工具。gRPC和REST各有优缺点，gRPC的proto文件需要客户端和服务器兼容性，而REST更具可发现性和灵活性。 - **recursivedoubts**：引用REST提出者Roy Fielding的观点，强调REST API应该让客户端只需知道初始URI，所有状态转换应由客户端选择，从而质疑gRPC的复杂性。 - **tyre**：简短提到REST是少数派。 - **swyx**：认为gRPC对于谷歌以外的开发者来说过于复杂和难以接近，特别是与REST相比。 - **Glyptodon**：质疑文章观点，并提到GraphQL，认为文章基于有限或不寻常的经验写成。 补充讨论： • gRPC的主要争议点在于其复杂性、调试困难、跨语言和团队使用的麻烦、以及与REST相比的开发体验。 • 支持者认为gRPC在内部服务通信、稳定性和性能上有优势，特别是通过工具和SDK生成可以解决许多问题。 • 中立观点则强调具体使用场景和需求，认为应根据项目实际情况选择合适的工具。

主要讨论点：关于网页存档工具的讨论，特别是Webrecorder和其它工具的有效性及使用体验 不同观点： • **barrettondricka的观点**： - 认为文章更像是在为Webrecorder做广告，而非实际的指南。 - 质疑文章末尾加入隐私和大科技公司部分的必要性。 - 指出文章中对工具功能的描述不够清晰，且提供的链接失效。 - 特别提到Webrecorder没有Firefox扩展，因此认为文章中关于隐私保护的讨论没有实际意义。 • **memalign的观点**： - 关注如何有效地存档网页应用程序，指出像Archivebox和archive.org这样的工具在捕捉动态加载的网页应用时效果不佳。 - 认为这个问题可能无法通过现有工具完全解决，可能需要手动工作和对网页应用代码的审计。 • **agieocean的观点**： - 提供了关于Archive.org的命令行接口（CLI）的有用链接，认为这对于对网页存档主题感兴趣的人是一个有价值的资源。 - 支持使用命令行工具来实现更复杂的存档任务。 补充讨论： • 争议的焦点在于Webrecorder的介绍是否过于广告化，以及文章中关于隐私保护的讨论是否恰当。 • 讨论中还涉及了现有网页存档工具在处理动态网页应用时的局限性，以及可能的解决方案，如手动干预和使用命令行工具。 • 具体工具的功能和链接失效问题也被提出，强调了用户在实际使用中的困扰。
