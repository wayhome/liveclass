文章摘要

MasterCard近期修复了一个域名服务器（DNS）配置错误，该错误可能导致其网络流量被拦截或重定向。这个错误是由于一个本应以“akam.net”结尾的DNS服务器域名被误配置为“akam.ne”，并持续了近五年。安全研究员Philippe Caturegli花费300美元注册了该域名，防止其被网络犯罪分子利用。他观察到每天有数十万DNS请求发送到他的服务器，表明不仅MasterCard，其他组织也犯了类似错误。虽然Caturegli本可以利用此漏洞进行恶意活动，但他选择了通知MasterCard。MasterCard确认问题但表示未对系统构成威胁，并修正了错误。Caturegli希望MasterCard能感谢或补偿他购买域名的费用，但MasterCard仅通过Bugcrowd计划间接表示其披露方式不符合道德安全实践。

Understanding gRPC, OpenAPI and REST and when to use them in API design

作者: hui-zheng | 评分: 217 | 评论数: 43 | 发布时间: 2025-01-23 00:47
文章摘要

本文讨论了API设计中的两种主要模型：RPC和REST，以及如何选择使用gRPC、OpenAPI和REST。gRPC是一种基于RPC模型的技术，使用HTTP/2作为传输协议，隐藏了HTTP的细节，客户端通过生成的存根进行调用。REST模型则强调通过服务器提供的URL进行访问，客户端无需了解URL格式。OpenAPI是另一种流行的API设计方式，客户端需要根据信息构建URL，与REST模型相反。本文为API设计者提供了选择这些技术的指导，指出REST适用于简单场景，而gRPC和OpenAPI更适合需要高效传输和明确结构的场景。

评论摘要
主要讨论点：gRPC与REST/OpenAPI的比较及其适用场景 不同观点： • **反对gRPC的观点**： - gRPC引发诸多技术难题，如调试困难、超时设置复杂、文档不完善等（jdwyah）。 - gRPC的工具链和依赖管理复杂，特别是在跨语言和跨团队合作时（bitzun, jon_richards, rednafi）。 - gRPC的客户端生成代码质量差，特别是在非主流语言中（gghoop, awinter-py）。 - gRPC隐藏了网络环境中的问题，如服务发现和认证，不如REST透明（rswail, gghoop）。 - gRPC不适合对外或跨语言的API，特别是在浏览器环境或COTS产品中（bitzun, toprerules）。 • **支持gRPC的观点**： - gRPC在内部服务通信中表现出色，特别是在大规模服务路由中（toprerules, rollulus）。 - gRPC的API规范更加简洁，只需定义“what”而无需关心具体实现细节（rollulus）。 - gRPC的工具和生态系统（如Buf）提供了向后兼容性和更少的讨论成本（rollulus）。 - gRPC生成的消息定义和响应使得API集成更稳定（abalaji）。 • **中立/混合观点**： - gRPC在某些场景下（如双向流通信）非常适合，但不适合所有场景（toprerules, jon_richards）。 - gRPC的性能优势在某些项目中未得到充分发挥，特别是在没有达到一定规模时（NAHWheatCracker）。 - OpenAPI和gRPC各有优劣，OpenAPI更适合描述HTTP API行为，而gRPC更适合高效的内部RPC通信（oppositelock）。 补充讨论： - 争议的焦点在于gRPC的复杂性与其带来的效益是否成正比。一些开发者认为gRPC在内部服务中表现出色，但在跨语言和对外API中表现不佳。 - 另一个争议点是gRPC的工具链和客户端生成代码的质量问题，特别是在非主流语言和跨团队合作时。 - REST的简单性和透明性被一些开发者认为是其最大的优势，特别是在需要与多种客户端和技术栈集成时。 - 对OpenAPI和REST的区分存在不同看法，有人认为OpenAPI只是REST API的描述工具，而gRPC则是一种RPC机制。 通过这些观点可以看出，gRPC和REST各有其适用场景，选择哪种技术取决于项目的具体需求和团队的技术栈。

Tech takes the Pareto principle too far

文章摘要

文章由游戏设计师Bobby Lockhart撰写，探讨了科技行业过度依赖帕累托原则（80/20法则）的问题。在游戏开发中，团队通过“垂直切片”展示游戏的完成度和开发能力，而科技行业则常依赖“最小可行产品（MVP）”来获取投资。MVP仅展示核心功能，无法证明团队能完成最终产品，而科技行业过于专注于快速原型设计，忽视了最后20%的精细打磨。文章指出，虽然帕累托原则在某些领域有效，但过度依赖会导致产品无法达到应有的完成度，尤其在AI等领域，当前方法可能无法实现最终目标。最后，作者认为社会若不过度关注帕累托原则，可能会更有利。

主要讨论点：围绕80/20原则及其在产品开发中的应用的讨论 不同观点： • **blululu的观点**：认为MVP（最小可行产品）和Vertical Slice（垂直切片）虽然在功能上相似，但目的不同。MVP旨在验证产品是否应被构建，而Vertical Slice用于竞争已有市场的产品。他以Magic Leap和Humane等硬件产品为例，指出长时间开发但市场不接受的风险，强调快速验证假设的重要性。 • **niemandhier的观点**：指出关键软件（如医疗设备控制、无人机飞行控制等）不应采用快速开发周期。他认为在不应使用这种方法的领域强行应用会导致问题，例如特斯拉的自动驾驶。他还对AI开发采用这种快速周期表示担忧。 • **lsy的观点**：指出80/20原则在工程中可以是有益的，例如解决80%用例但只花费20%成本的产品。然而，当产品在所有任务中只有80%的可靠性时，用户难以依赖，因为无法建立对产品工作原理的稳定心理模型。 • **shswkna的观点**：强调最后的20%努力虽然只产生20%的成果，但提供了完成感和价值感，这种感受是无价的。完成一件有价值的事情带来的满足感和成就感是无法量化的。 • **bigmattystyles的观点**：认为每个概念都有被过度使用的风险。他分享了几个原则，如“不要让完美成为优秀的敌人”和“低承诺高实现”，以避免在开发中陷入困境。 • **unsigner的观点**：指出游戏行业也存在对原型过度优化的问题，Unity和Unreal引擎的普及部分原因在于此。 • **openrisk的观点**：认为“垂直切片”概念的普及是有价值的，特别是在思考如何构建1.0版本的交付物时。 horizontal和vertical切片的区分有助于理解不同开发策略。 • **knowhy的观点**：质疑80/20原则的普遍适用性，认为它被误用为社会阶层合理化的工具。他指出并不是所有分布都遵循80/20原则，且这种观念可能导致不必要的社会不平等。 • **scarab92的观点**：认为如果MVP找到了产品市场契合点且市场足够大，经济激励会促使完成剩余的20%工作。如果市场不够大，客户仍然得到了80%的价值，比没有产品要好。 • **gunian的观点**：以一个假设情景提问，如果生命只剩8-12周且项目只完成了5%，应如何分配资源。 • **emsal的观点**：认为对80/20原则的讨论存在误解，80%和20%通常指的是经济系统中竞争的代理人/生产者/客户的比例，而不是指开发过程中的努力分配。 • **renewiltord的观点**：表示喜欢阅读半成品开源软件，因为可以获得很多想法。 • **wickedsight的观点**：认为80%的AI技术可能已经对大部分人足够有用，即使对最顶尖的20%人群来说不够好，但对大部分人来说，80%已经提供了巨大价值。 • **awesome_dude的观点**：认为大多数人并不在乎产品是否完美，只要产品能完成基本任务即可。他以多种日常用品为例，表示自己愿意使用不完美的产品。 补充讨论： - 关于80/20原则的适用范围和误解是讨论的核心，特别是在产品开发和关键系统中的应用。 - 争议的焦点在于快速开发和验证方法在不同领域的适用性，尤其是对关键软件和AI开发的担忧。 - 完成感和价值感在产品开发中的重要性也被多次提到，强调了无形的成就感。
