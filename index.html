Homomorphic Encryption in iOS 18

作者: surprisetalk | 评分: 173 | 评论数: 11 | 发布时间: 2025-01-11 16:35
文章摘要

本文介绍了iOS 18中使用的同态加密技术，旨在让普通用户更容易理解苹果如何在保护隐私的同时改进照片应用的搜索功能。文章解释了如何将图片内容表示为数值向量（嵌入向量），并通过算法（如余弦相似度）进行比较，以找到相关图片。为了保护用户隐私，这些计算本应在设备上完成，但随着数据库增长，设备无法存储所有数据。 同态加密是解决方案：它允许在服务器上处理加密数据而不泄露原始内容。服务器可以在不知道数据具体值的情况下执行操作（如加法和乘法），处理后的数据仍保持加密状态，只有用户能解密并获得结果。这种技术复杂且仍在发展中，但为隐私保护提供了强有力的工具。文章最后呼吁更多密码学专家参与完善这一技术。

Don't use cosine similarity carelessly

作者: stared | 评分: 266 | 评论数: 16 | 发布时间: 2025-01-14 21:23
文章摘要

文章主要讨论了在使用向量和余弦相似性（cosine similarity）时需要谨慎，不能盲目应用。尽管向量在人工智能中非常有用，可以捕捉实体之间的关系，但简单地使用余弦相似性可能会导致误导性结果。文章通过例子说明，余弦相似性可以有效区分语义相近的句子，如"Python可以让你致富"和"精通Python可以装满你的口袋"（相似度0.750），而不是仅看字面相似性如"Python可以让你致富"和"Python可以让你发痒"（相似度0.576）。然而，余弦相似性有其局限性，它像胶带一样，虽然方便但不能解决深层次问题。文章提醒数据科学家在使用余弦相似性时要更加有意图和谨慎，以获得更好的结果。

评论摘要
主要讨论点：RAG（检索增强生成）应用中的相似性度量和重排方法 不同观点： • **Pamelafox** 认为，在RAG应用中使用余弦相似度进行检索后，应使用“语义重排器”或“L2重排模型”对结果进行重排，以更好地匹配用户查询。她提到可以使用交叉编码器模型进行重排，并分享了具体的GitHub示例。此外，她还提到在Azure中使用AI Search的语义排序器进行重排。 • **Bambax** 对文章中关于余弦相似度的例子提出质疑，认为直接使用LLM进行查询比较可能会得到相同或更差的结果。他指出，LLM在比较句子相似性时，可能会与余弦相似度结果接近。 • **Bjourne** 讨论了词向量的局限性，指出由于维度灾难问题，词向量的效果可能不佳。他提出了一种使用加权最短路径计算词相似性的方法，虽然速度较慢，但可能更有用。 • **Deepsquirrelnet** 支持使用交叉编码器进行重排，认为这是高性能且更快的解决方案，并分享了在Hugging Face上发布的基于ModernBERT的STS交叉编码器。 • **Romanhn** 提出了关于如何处理查询中排除条件的问题，例如“关于旧金山的文章且不提及汽车”，并询问在余弦相似度下如何处理这种排除条件。 • **Cranium** 提到HyDE（假设文档嵌入）可以在上下文不明确的情况下表现更好。通过让LLM生成假设答案并使用该答案进行检索，可以避免直接嵌入用户问题导致的检索不准确。 • **Visarga** 提出了一种使用LLM生成标题、摘要、关键词列表、主题等信息，然后将这些信息组合嵌入的方法，以解决余弦相似度仅在表面层次工作的问题。 • **Weitendorf** 认为余弦相似度和top-k RAG方法较为原始，提出需要通过在训练中结构化模型来解决RAG问题，而不是依赖于模型外部的启发式方法和阈值。 补充讨论： • **Nomilk** 提到在描述著名引言时，LLM往往无法找到正确答案，可能是因为描述中使用了“类似”的词语而非确切词语。 • **Mlepath** 指出在机器学习中，任何相似性度量都有其优缺点，文章中推荐的点积相似度也有一些问题，如未归一化、偏向大向量等。 • **Pooyak** 询问是否存在可以根据系统提示生成嵌入的模型，以便在相似性搜索和聚类中使用，而无需进行太多自定义工作。 争议焦点： • **余弦相似度的有效性**：一些评论者（如Bambax）质疑余弦相似度在比较句子相似性方面的效果，认为直接使用LLM进行比较可能更好，而另一些评论者（如Pamelafox和Deepsquirrelnet）则支持使用语义重排器或交叉编码器进行重排以提高准确性。 • **重排方法的选择**：不同评论者提出了不同的重排方法，包括语义重排器、L2重排模型、交叉编码器等，每种方法都有其支持者和适用场景。 其他值得注意的讨论点： • **处理排除条件**：Romanhn提出的如何在查询中处理排除条件的问题，例如在检索中如何避免提及某些内容。 • **维度灾难问题**：Bjourne提出的词向量由于维度灾难问题效果不佳，建议使用加权最短路径方法计算词相似性。 • **模型训练和结构化**：Weitendorf提出通过在训练中结构化模型来解决RAG问题，而不是依赖外部启发式方法。 总结：评论中讨论了RAG应用中余弦相似度及其替代方法的有效性，重点讨论了重排方法的选择和处理排除条件的问题，并提出了一些改进建议和替代方案。

Distributed Transactions at Scale in Amazon DynamoDB

作者: lambrospetrou | 评分: 16 | 评论数: 1 | 发布时间: 2025-01-12 11:12
文章摘要

Show HN: Pyper – Concurrent Python Made Simple

文章摘要

Pyper 是一个简化并发和并行数据处理的 Python 框架，基于函数式编程模式，适用于 ETL 系统、数据微服务和数据采集等场景。其主要特点包括： 1. **直观的 API**：易于学习和理解，统一了线程、多进程和异步任务的抽象。 2. **函数式范式**：使用 Python 函数构建数据管道，代码可重用性强。 3. **安全性**：隐藏底层任务执行和资源清理的复杂性，避免竞态条件、内存泄漏等问题。 4. **高效性**：采用队列、工作线程和生成器的延迟执行设计。 5. **纯 Python 实现**：轻量，无额外依赖。 通过 `task` 装饰器，用户可以定义并发任务管道，使用 `|` 操作符连接不同任务，支持异步和同步任务的混合使用。Pyper 自动处理并发和并行机制，开发者只需关注业务逻辑。
